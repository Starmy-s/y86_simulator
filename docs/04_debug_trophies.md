# 🛠️ 04. 功勋墙与仿真避坑指南

在计算机体系结构的仿真开发中，跨越“书本理论”到“工程落地”往往需要经历真枪实弹的 Debug 锤炼。硬件电路是天然并行的，而高层软件语言（如 C++）则存在着严格的单线程隐式时序和语言边界安全限制。这两者的物理碰撞，会衍生出许多极其隐蔽、让初学者痛不欲生的“毁灭性天坑”。

本手册白纸黑字地记录了本项目在从零到完整通电的过程中，遭遇并踩掉的三个最为经典的“硬件仿真坑”。将其公诸于此，作为功勋墙，供后来者引以为戒，少走弯路。

## 🔥 坑 1：类成员声明顺序引发的内存暴死 (`std::bad_alloc`)

### 1. 事故现场

在虚拟微处理器刚通电跑起来的瞬间，系统没有任何征兆地当场崩溃，控制台甩出一行惨烈的报错：

```bash
terminate called after throwing an instance of 'std::bad_alloc'
  what():  std::bad_alloc
Aborted (core dumped)

```

排查发现，崩溃点竟然发生在内存模块（`Memory`）的初始化构造阶段。

### 2. 微架构根源（硬件思维 vs C++ 隐式时序）

在硬件逻辑中，主板内存的大小 `size` 和物理存储阵列 `storage` 是并存的。在编写 C++ 构造函数的初始化列表时，很多初学者会极其自然地写出以下代码：

```cpp
// 表面上看起来无懈可击的构造函数
class Memory {
private:
    std::vector<uint8_t> storage; // 物理存储阵列
    uint64_t size;                // 内存字节大小

public:
    Memory(uint64_t bytes) : size(bytes), storage(size, 0) {} 
    // 心理预期：先用 bytes 赋值给 size，再用 size 去初始化 vector 的大小
};

```

然而，**C++ 语法规定：类成员变量在初始化列表中的物理执行顺序，严格按照它们在 `private:` 树状结构中的声明顺序执行，与你在构造函数冒号后面写的先后顺序毫无关系！**

在上面的代码中，`storage` 被声明在 `size` 的前面。因此，当 CPU 执行构造函数时，它会**先去初始化 `storage**`。此时，用来决定 vector 大小的变量 `size` 还没有被赋值，它里面躺着的只是内存里的**垃圾随机值**（通常是一个极大的无符号巨型数字）。`std::vector` 傻傻地拿着这个随机的几千万 GB 的大小试图去向操作系统申请物理内存，瞬间被 Linux 内核无情熔断，甩出 `bad_alloc` 崩溃暴死。

### 3. 本项目优雅解法

彻底解耦成员变量之间的初始化依赖。在初始化列表中，**绝不使用类内变量互调**，而是统一使用外来的传参形参（`bytes`）直接干净地初始化所有底层实体：

```cpp
// 本项目对准硬件的规范写法
class Memory {
private:
    std::vector<uint8_t> storage; // 声明顺序不再成为定时炸弹
    uint64_t size;

public:
    Memory(uint64_t bytes) : size(bytes), storage(bytes, 0) {} 
    // 统一使用物理形参 bytes，彻底斩断时序依赖
};

```

## 🔥 坑 2：总线接收载体写错导致的物理“高位截断”

### 1. 事故现场

测试用例中明明有一行汇编指令 `irmovq $0x1234, %rax`，预期是将 `0x1234` 灌入寄存器。但当程序运行完毕、dump 出寄存器现场时，发现 `%rax` 里存着的居然只有 `0x0000000000000034`！高位的 `12` 在传输过程中神秘蒸发。

### 2. 微架构根源（细水管装不下物理洪水）

在取指阶段（Fetch），CPU 需要从机器码字节流缓冲区中捞出 8 字节宽度的立即数（`valC`）。为了追求极致的高效，本项目硬核地利用了宿主机（X86/ARM）同为小端序的物理特征，直接采用了高效的**小端序指针强转直接拷贝法**：

```cpp
// 取指阶段的内部总线临时逻辑
uint8_t icode = (storage[pc] >> 4) & 0x0F;
// ... 经过复杂的 MUX 判断需要捞取立即数
uint8_t valC = *(uint64_t*)&storage[pc + 2]; // 👈 凶手就在这里！

```

看出了吗？右边的指针强转 `*(uint64_t*)&` 惊艳绝伦，它确实瞬间从内存槽里完整地搂出了 8 个字节的干净大整数 `0x0000000000001234`。但是，左边用来接收这条数据总线信号的临时载体，居然手软被写成了 **`uint8_t valC`**（只有 1 个字节宽度）！

这根 1 字节宽的细水管根本装不下 8 字节的数字洪水。C++ 编译器在底层默默地执行了隐式无符号截断，把高位的 7 个字节无情咔嚓掉，只留下了最低位的 `0x34` 投递给后续的 ALU 总线，造成了惨烈的硬件级信息降维打击。

### 3. 本项目优雅解法

全面清查并打通虚拟主板上的所有 64 位宽内部数据总线，将所有涉及 `valA`, `valB`, `valC`, `valE`, `valM` 的临时总线载体类型，严密更正、对齐为 **`uint64_t`**：

```cpp
// 规范打通 64 位物理大总线
uint64_t valC = *(uint64_t*)&storage[pc + 2]; // 载体升级，完美并联

```

## 🔥 坑 3：取指边缘内存不足 10 字节引发系统“暴死”

### 1. 事故现场

当编写的测试 Y86-64 汇编程序运行到最后一条指令 `halt`（机器码 `00`，物理地址位于内存的最末尾）时，模拟器非但没有体面地打印停机报告，反而当场爆出 `std::out_of_range` 越界异常，程序直接“猝死”。

### 2. 微架构根源（鲁莽取指 vs 内存物理边界）

体系结构教科书上说：Y86-64 的指令最长是 10 个字节（如 `irmovq`）。所以在SEQ（单周期）微架构的取指硬件设计中，译码器不管三七二十一，在当前的 `pc` 周期开闸时，会硬性地向后连搂 10 个字节扔进临时缓冲区，然后再通过 `icode` 去细分到底裁切多长。

在硬件电路中，如果 `pc` 到了内存最后 1 个字节，后面已经没有物理电容阵列了，取指总线捞上来的只会是空电平零（`0x00`）。但我们在 C++ 里是用 `std::vector` 模拟内存的，**当你直接使用 `storage[pc + 9]` 去强行读取不存在的越界槽位时，标准库会当场触发越界保护，直接把你的仿真进程彻底杀掉**！

```cpp
// 鲁莽的单周期取指行为
uint64_t next_pc = pc;
uint8_t code_byte = storage.at(pc); 
// 如果 pc 已经是最后一个字节，下面强行读取 pc+1 到 pc+9 的行为会直接引发 vector 暴死

```

### 3. 本项目优雅解法

在 `Memory` 模块的底层原语中，拒绝使用鲁莽的直接下标访问，而是引入 **`std::min` 物理长度限制保护与安全补零机制**。

当取指总线发现当前 `pc` 距离主存物理边界已经不足 10 字节时，它会极其聪明地“能拿几字节就拿几字节”，剩下拿不到的空槽位，统一在软件缓冲区中**安全补零（`0x00`）**。

```cpp
// 本项目安全、高仿真的边缘取指原语实现
void read_bytes_safe(uint64_t start_pc, uint8_t* dest_buffer) {
    // 1. 严密死守内存物理边界，算出当前最多还能捞多少字节
    uint64_t available_bytes = (start_pc < size) ? (size - start_pc) : 0;
    uint64_t bytes_to_read = std::min(available_bytes, uint64_t(10));
    
    // 2. 将安全的合法内存拷贝进缓冲区
    if (bytes_to_read > 0) {
        std::memcpy(dest_buffer, &storage[start_pc], bytes_to_read);
    }
    
    // 3. 核心补零机制：如果不足 10 字节，用硬件零电平（0x00）喂饱 CPU 的取指分析器
    if (bytes_to_read < 10) {
        std::memset(dest_buffer + bytes_to_read, 0, 10 - bytes_to_read);
    }
}

```

这样一来，哪怕 `halt` 指令孤独地躺在物理主存的最后一个字节，模拟器也能安全地抓取到它，并用后面补出来的 `0x00` 电平完美骗过译码器，让 CPU 最终体面、高仿真地安全退场！
