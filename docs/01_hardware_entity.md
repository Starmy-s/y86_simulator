# 💾 01. 硬件实体转世手册

在计算机体系结构中，行为级模拟器（Behavioral Simulator）的核心目标是在高层软件语言（如 C++）中，复现处理器内部**状态（State）与功能（Function）的绝对等价**，而不是在晶体管或门电路层面进行物理等价。

为了在软件世界中搭建起这块数字化主板，我们需要将硬件电路解构。在 Y86-64 处理器内部，所有元器件可以被严密地划分为 **4 大底层物理阵列**。本手册将为你详细清点这些硬件实体在 C++ 代码中的虚拟转世与时序对齐方式。

---

## 一、 存储型硬件（Storage Elements）

这类硬件在数字电路中由**时序逻辑电路（Sequential Logic）**构成（通常是 D 触发器物理阵列）。它们具备物理状态的维持能力，在单周期（SEQ）微架构内严格表现为**“期初读、期末写”**。

为了在 C++ 中彻底消除软件代码“从上到下执行”的时间先后顺序污染、完美复现时钟上升沿的合闸锁存，项目在类内统一采用了**双缓冲状态覆盖**机制。

### 1. 通用寄存器堆 (Register File)

* **真实硬件**：由 15 个 64 位 D 触发器组构成的硬件阵列，拥有两个独立的读端口（Read Ports）和一个写端口（Write Port），支持在单个时钟周期内同时读取 `rA`、`rB` 引脚的数据，并在时钟沿到来时改写目标寄存器。
* **C++ 实现深度 (100% 状态对齐)**：
在 CPU 类中，我们使用一个固定大小的静态数组来模拟这个黄金抽屉阵列：
```cpp
private:
    uint64_t registers[15] = {0}; // 物理映射 %rax (0) 到 %r14 (14)

```


* **读写时序对齐**：
* **读模拟**：在译码阶段，直接通过数组下标任意读取（如 `registers[srcA]`），其本质等价于多路复用引脚引出电平，瞬间完成，无状态改写。
* **写模拟**：引入 `next_registers` 预写缓冲数组。所有写回阶段（Writeback）的计算成果只允许投递给缓冲。只有当周期结束、时钟上升沿真正到来时，才执行统一刷新，完美复现**单时钟沿同步锁存**。



### 2. 程序计数器 (Program Counter - PC)

* **真实硬件**：一个独立的 64 位时序寄存器，死死指着当前或下一条待执行指令的内存物理地址。它的输出引脚直接物理连接在指令主存的地址译码器上。
* **C++ 实现深度 (100% 时序对齐)**：
```cpp
private:
    uint64_t pc = 0; // 虚拟微处理器的 PC 指针

```


* **控制闸门对齐**：严格禁止在取指或执行阶段直接修改 `pc` 的值。在各阶段代码运行过程中，所有计算出的跳转或顺序推进地址只能暂存在局部总线变量 `next_pc` 中。必须在整个时钟周期（`clock_tick` 函数）的最后一行代码执行 `pc = next_pc;`。这就精准对齐了硬件电路在电平跳变那一瞬间的锁存时序。

### 3. 条件码寄存器 (Condition Codes - CC)

* **真实硬件**：由 3 个独立的 1 位 D 触发器（ZF-零标志, SF-符号标志, OF-溢出标志）组成的寄存器阵列。它们就像汽车仪表盘上的三个红绿警报灯，只在执行算术逻辑运算（`opq`）时由控制信号决定是否更新。
* **C++ 实现深度 (100% 逻辑对齐)**：
我们使用一个独立的布尔结构体来模拟这三根物理标志线：
```cpp
struct ConditionCodes {
    bool ZF = 1; // Zero Flag：上次计算结果是否为 0
    bool SF = 0; // Sign Flag：上次计算结果是否为负数
    bool OF = 0; // Overflow Flag：上次计算是否发生补码溢出
};
ConditionCodes cc;

```


逻辑完全对齐：只有当控制逻辑块发出 `set_cc` 使能信号（高电平）时，判定电路才会改写下一周期的 CC。其他传送或访存指令通过时，这三个灯泡雷打不动，完美保持原状态。

## 二、 计算与组合逻辑硬件（Combinational Hardware）

这些硬件在芯片内部由纯粹的**门电路（门阵列、数据选择器、加法器等）**构成。它们最大的特点是**没有时钟线接通**，其内部不维护任何历史状态，**电信号喂进去，受限于物理导线延迟，极短时间内立刻自发吐出结果**。在 C++ 代码中，我们将其**完全剥离成“纯函数（Pure Functions）”**。

### 4. 算术逻辑单元 (ALU)

* **真实硬件**：由半加器、全加器、位移器以及庞大的与或非门阵列构成的核心计算网格。它的输入端死死接收数据输入 `A` 和 `B` 以及操作控制信号 `alu_fun`，其输出端在吐出计算成果 `valE` 的同时，还会实时拉出 3 根状态导线（硬件引脚）送往上面的条件码判定电路。
* **C++ 实现深度 (功能级 100% 对齐)**：
在行为级仿真中，我们不需要用布尔代数去拼电路门，而是直接利用宿主机（你的 Intel/AMD/Apple 芯片）的 CPU 本身的算术运算符进行**数学等价模拟**：
```cpp
uint64_t run_alu(uint8_t alu_fun, uint64_t valA, uint64_t valB, ConditionCodes& next_cc, bool set_cc) {
    uint64_t valE = 0;
    switch (alu_fun) {
        case A_ADD: valE = valB + valA; break; // 利用宿主机硬件加法模拟 Y86 加法
        case A_SUB: valE = valB - valA; break;
        case A_AND: valE = valB & valA; break;
        case A_XOR: valE = valB ^ valA; break;
    }

    if (set_cc) {
        next_cc.ZF = (valE == 0);
        next_cc.SF = ((valE >> 63) & 1);
        // 补码溢出判定组合逻辑
        if (alu_fun == A_ADD) {
            next_cc.OF = (((valA >> 63) == (valB >> 63)) && ((valE >> 63) != (valA >> 63)));
        } else if (alu_fun == A_SUB) {
            next_cc.OF = (((valA >> 63) != (valB >> 63)) && ((valE >> 63) != (valB >> 63)));
        }
    }
    return valE;
}

```



### 5. 字节拆分器 (Splitter) / 符号扩展器 (Sign Extender)

* **真实硬件**：
* **Splitter**：在硅片上甚至不占用任何物理门，就是纯粹把 8 根平行的铜总线导线一分为二，4 根拉走当指令大闸（`icode`），4 根拉走当功能控制线（`ifun`）。
* **Sign Extender**：把机器码里抽出来的短字节地址或立即数，通过硬连线复制其最高位（符号位），扩展成标准的 64 位补码。


* **C++ 实现深度 (硬连线等价 100%)**：
* 在 C++ 中通过极致高效的**位移和掩码**模拟 Splitter：`icode = (code_byte >> 4) & 0x0F; ifun = code_byte & 0x0F;`。
* 通过 C++ 的强类型转换或位宽填充技术模拟符号扩展器，直接利用编译器的底层技术实现高位符号位的自发填充。

## 三、 传输与互联硬件（Interconnect & Bus）

### 6. 数据总线与地址总线 (Buses) / 多路选择器 (MUX)

* **真实硬件**：
* **Buses**：布满主板和芯片内部的密集铜导线走线，负责在各个微架构阶段之间投递高低电平信号（如指令线 `valC`、读出总线 `valA`/`valB`、访存总线 `valM` 等）。
* **MUX (数据选择器)**：纯硬件开关门，根据控制块提供的使能信号，决定让哪一根导线上的电平数据流过。例如：`srcA` 控制信号决定了 `valA` 通路当前连接的是 `%rax` 的引脚，还是 `%rsp` 的引脚。


* **C++ 实现深度 (纯抽象等价)**：
* **总线模拟**：项目**不需要**在 C++ 里声明复杂的物理导线对象，而是直接利用各个阶段函数内部的局部变量（Local Variables）来扮演总线（如 `uint64_t valA`）。这些局部变量在生命周期（当前函数/周期循环）结束时自动销毁，完美对应了硬件中“电平信号随着时钟周期结束而瞬间消失”的物理特性。
* **MUX 选路模拟**：直接用 C++ 的**三目运算符**或 `switch-case` 开关语句模拟硬件的多路选通：
```cpp
// 硬件 MUX 开关切换：如果源引脚有效则接通寄存器，否则将总线拉低为 0 电平
uint64_t valA = (srcA != R_NONE) ? registers[srcA] : 0;

```

## 四、 系统边界与核心指挥官（Boundaries & Commander）

### 7. 物理内存条 (Main Memory / RAM)

* **真实硬件**：独立坐落在 CPU 外部主板芯片组上的大容量物理电容阵列（或高速 SRAM 缓存），通过主板上的物理引脚与 CPU 总线连接。
* **C++ 实现深度 (黑盒解耦模拟 90%)**：
在代码中通过封装了边界拦截的 `std::vector<uint8_t>` 模块进行全黑盒模拟。
* **对齐部分**：严密模拟了内存的**物理边界检查**（越界直接拉高异常状态线，抛出 `S_ADR` 熔断信号），以及小端序（Little-Endian）的字节流拼接拆解（8 字节的长整型数据如何有规律地分布在 8 个连续内存槽位里）。
* **忽略部分**：作为单周期行为级模拟，我们目前**剥离了真实的物理延迟（Latency）与电荷刷新时间**。在当前设计中，访存操作被视作瞬间完成。



### 8. 控制逻辑块 (Control Logic Blocks)

* **真实硬件**：这是 CPU 内部最烧脑、最密集的纯组合逻辑硬核电门。它们不维护任何状态，只负责盯着当前取出的指令大闸 `icode`、功能分支 `ifun` 以及三个警报灯 `CC`。通过布尔代数化简出来的门电路，实时向全机吐出各种 0 或 1 的使能开关信号（如 `need_valC`、`srcA`、`set_cc`、`instr_valid`）。
* **C++ 实现深度 (100% 逻辑表达式对齐)**：
在代码中，我们拒绝编写面条式的、带有大量主观嵌套的过程代码，而是将其编写为**与硬件真值表、布尔逻辑表达式完全一一对应的纯布尔判断函数**：
```cpp
// 核心控制块电门：只有当 icode 为 OPQ（算术运算）时，才会将 set_cc 导线拉高为 1
bool ctrl_set_cc(uint8_t icode) {
    return (icode == I_OPQ); 
}

// 核心控制块 MUX 开关：决定写回阶段的数据源来自 ALU (valE) 还是来自内存 (valM)
uint8_t ctrl_dstE(uint8_t icode, uint8_t rB, bool cnd) {
    if (icode == I_RRMOVQ && cnd) return rB; // 条件满足时，选通 rB 写回
    if (icode == I_IRMOVQ || icode == I_OPQ) return rB;
    if (icode == I_PUSHQ  || icode == I_POPQ || icode == I_CALL || icode == I_RET) return R_RSP;
    return R_NONE; // 物理断路，不改写任何寄存器
}

```
